## 1. Введение
Данный мессенджер представляет собой распределенную систему обмена мгновенными сообщениями, ориентированную на приватность и высокую доступность. В отличие от стандартных решений, архитектура строится на модели "Zero-Trust", где сервер выполняет роль транспорта, не имея доступа к открытому тексту переписки.

## 2. Архитектура системы
Система состоит из трех ключевых слоев:
*   **Клиентский слой:** Реализован на базе кроссплатформенных фреймворков, поддерживает локальную базу данных (SQLite/SQLCipher) для хранения кэша сообщений.
*   **Серверный слой (Gateway/API):** Обрабатывает входящие подключения, балансирует нагрузку и управляет очередями.
*   **Слой хранения (Storage Layer):** Использует распределенные БД для обеспечения высокой консистентности.

## 3. Протоколы передачи данных
*   **Транспортный уровень:** Основной обмен ведется через WebSockets (WSS). В случае нестабильного соединения система переключается на Long Polling.
*   **Сериализация:** Вся передача данных осуществляется через **Protocol Buffers (Protobuf)**. Это позволяет минимизировать размер пакетов и повысить скорость десериализации на клиенте.
*   **Структура пакета:** Каждый пакет содержит `header` (тип операции, версия, ID сессии) и `payload` (зашифрованное тело).

## 4. Механизмы сквозного шифрования (E2EE)
Это ядро безопасности системы. Реализация основана на протоколе Double Ratchet:
*   **Первичный обмен:** Используется алгоритм Диффи-Хеллмана на эллиптических кривых **Curve25519**.
*   **Double Ratchet:** Позволяет менять ключи шифрования для каждого отдельного сообщения (Perfect Forward Secrecy). Если ключ одного сообщения будет скомпрометирован, это не даст доступа к прошлым или будущим сообщениям.
*   **Алгоритм шифрования:** AES-256 в режиме GCM (Galois/Counter Mode) для обеспечения целостности и конфиденциальности.
*   **Хеширование:** SHA-256 используется для создания fingerprint'ов ключей, которые пользователи могут сравнивать для подтверждения идентичности собеседника.

## 5. Алгоритмы аутентификации
Процесс включает несколько этапов:
1.  **Регистрация:** Клиент генерирует пару ключей (Public/Private), публичный ключ отправляется на сервер как идентификатор пользователя.
2.  **Аутентификация:** Использование JWT с коротким временем жизни (15 минут).
3.  **Обновление сессии:** Refresh-токен хранится в зашифрованном виде (Secure Storage) на устройстве.

## 6. Управление сессиями и WebSocket
Сервер поддерживает таблицу активных сессий в **Redis**.
*   **Heartbeat (Ping/Pong):** Клиент отправляет ping каждые 30 секунд. Если сервер не получает ответ в течение 90 секунд, сессия помечается как `Disconnected`.
*   **Multi-device:** Система поддерживает до 5 активных устройств на один аккаунт. Синхронизация между ними происходит через зашифрованные "сервисные сообщения".

## 7. Хранение данных и репликация
*   **Hot Data:** Последние 100 сообщений хранятся в оперативной памяти (Redis).
*   **Cold Data:** Вся история записывается в шардированную базу данных (PostgreSQL), где каждый пользователь является отдельным шардом.
*   **Удаление:** Реализована функция "Удалить у всех", которая отправляет специальный пакет `DELETE_MESSAGE` получателю, инициируя удаление из локальной БД клиента.

## 8. Процесс доставки сообщений
1.  **Отправка:** Клиент шифрует сообщение ключом, полученным через рукопожатие.
2.  **Валидация:** Сервер проверяет ID отправителя и получателя.
3.  **Маршрутизация:** Проверка Redis на наличие `ConnectionID` получателя.
4.  **Статусы:** 
    *   `Sent`: Сервер принял пакет.
    *   `Delivered`: Пакет дошел до устройства получателя.
    *   `Read`: Клиент отправил сигнал о прочтении (зашифрованный).

## 9. Синхронизация состояний
При подключении клиент запрашивает `LastSequenceID`. Сервер отдает все сообщения, начиная с этого ID. Это обеспечивает отсутствие потерь данных даже при временных разрывах связи.

## 10. Обработка вложений и медиа
*   Файлы загружаются на S3-совместимое хранилище.
*   Перед загрузкой файл шифруется на стороне клиента с использованием ключа, который передается получателю в отдельном, зашифрованном сообщении. Сервер хранит только "зашифрованный мусор".

## 11. Безопасность и угрозы
*   **MITM (Man-in-the-middle):** Невозможен благодаря принудительному TLS 1.3 и проверке ключей.
*   **DoS:** Лимит на количество WebSocket-соединений с одного IP-адреса.
*   **Локальные данные:** Весь локальный кэш зашифрован ключом, который хранится в системном KeyChain (iOS) или Keystore (Android).

## 12. Инфраструктура и масштабируемость
Система развернута в контейнеризированной среде (Kubernetes). Горизонтальное масштабирование осуществляется путем добавления новых узлов Gateway, которые связаны через общую шину данных (NATS/Kafka).
