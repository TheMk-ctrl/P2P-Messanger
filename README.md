# Nopeer.pw

> **Zero-Server, Peer-to-Peer Encrypted Web Messenger**

`Nopeer.pw` — это бессерверный веб-мессенджер, работающий прямо в браузере и обеспечивающий сквозное шифрование (E2E) сообщений, файлов и медиапотоков. Проект исключает централизованный сервер хранения сообщений: все данные передаются напрямую между участниками с использованием протокола WebSockets (сигнализация) и WebRTC (p2p-канал).

## 🏗 Архитектура и ключевые особенности

Проект построен по принципу **Zero-Knowledge / Serverless Messaging**:
* **Отсутствие базы данных сообщений:** Ни одно сообщение, имя пользователя или ключ шифрования никогда не сохраняются на центральном сервере.
* **Direct Peer-to-Peer:** После установки соединения сервер сигнализации выбывает из цепочки обмена данными.
* **Клиентская обработка:** Вся логика шифрования, рендеринга Markdown и управления состоянием выполнена на стороне клиента (Web API, Web Crypto API).
* [ Client A ]                                               [ Client B ]
(Browser)                                                  (Browser)
│                                                          │
├─────────────► [ Signaling Server (PeerJS) ] ◄────────────┤ (Установление p2p)
│                                                          │
============================================================
│               Прямое P2P-соединение (DataChannel)        │
│ ◄──────────────────────────────────────────────────────► │ (Зашифрованные сообщения)

## 🛠 Подробный разбор функционала

### 1. Генерация и управление комнатами (Session Manager)
* **Создание сессии:** При входе генерируется уникальный идентификатор Peer ID (или используется заданный пользователем).
* **Присоединение по ссылке / QR-коду:** Модуль формирует прямой URL с хешем сессии, позволяя второму пиру подключиться в один клик.

### 2. P2P Текстовый чат и форматирование (Messaging Engine)
* **Поддержка Markdown:** Входящие и исходящие текстовые блоки парсятся на лету, позволяя использовать форматирование текста, таблицы, списки и блоки кода.
* **Статусы сообщений:** Отслеживание состояний доставки (`Отправлено`, `Доставлено`, `Прочитано`) через подтверждающие P2P-пакеты (ACK).

### 3. Передача файлов и медиа (Chunked Binary Transfer)
* **Передача файлов любого типа:** Изображения, документы, архивы передаются напрямую от клиента к клиенту.
* **Progress Bar:** Отображение реального прогресса передачи и скорости на основе полученных фрагментов (чанков).

### 4. Голосовые и видеовызовы (WebRTC MediaStreams)
* **Аудио-вызовы:** Использование `getUserMedia` API для захвата и потоковой передачи звука без промежуточных медиасерверов (SFU/MCU).

## 🔒 Безопасность и сквозное шифрование (E2EE)

Безопасность данных в **Nopeer.pw** гарантируется применением **Web Crypto API** (нативный симметричный и асимметричный криптографический стек браузера).

### Алгоритм шифрования и ключи

1. **Генерация пар ключей (Diffie-Hellman / ECDH):**
   * При старте сессии каждый клиент создает временную пару ключей по алгоритму **ECDH (Elliptic Curve Diffie-Hellman)** на кривой **P-256** (или X25519).
   * Публичные ключи обмениваются через рукопожатие на этапе установления соединения.

2. **Согласование общего секрета (Shared Secret):**
   * Клиент A использует закрытый ключ $K_{privA}$ и публичный ключ $K_{pubB}$.
   * Клиент B использует закрытый ключ $K_{privB}$ и публичный ключ $K_{pubA}$.
   * Оба клиента получают одинаковый общий секрет $S$, не передавая его по сети.

3. **Симметричное шифрование (AES-GCM 256-bit):**
   * На основе общего секрета $S$ через функцию KDF (HKDF-SHA256) вычисляется мастер-ключ шифрования.
   * Каждое сообщение и файл шифруются алгоритмом **AES-GCM-256**.
   * Для каждой отдельной операции генерации шифротекста используется случайный уникальный **IV (Initialization Vector / Nonce)** длиной 96 бит.

+-------------------------------------------------------------------+
|                        Структура Пакета                           |
+------------------------------------+------------------------------+
| IV (12 bytes)                      | Encrypted Payload + Auth Tag |
+------------------------------------+------------------------------+


4. **Аутентификация данных (AEAD):**
   * Использование режима GCM обеспечивает проверку целостности (Authentication Tag). Любое изменение шифротекста в процессе передачи приведет к ошибке расшифровки на стороне получателя.

## 🌐 Сетевое взаимодействие и передача данных

### Signaling (Сигнализация)
Для первоначального обмена сетевыми адресами (ICE candidates) и ключами ECDH используется легкий сигнальный сервер на базе **Node.js + PeerJS**.
* Сервер **не имеет доступа** к текстовому содержимому сессий.
* Сервер видит только факты подключения Peer ID A к Peer ID B.

### WebRTC DataChannel (Обмен данными)
После завершения сигнальной фазы создается `RTCDataChannel`:
* **Опции канала:** `ordered: true` (гарантированный порядок доставки пакетов).
* **Передача файлов:**
  1. Файл разбивается на чанки размером 16–64 KB (в зависимости от буфера).
  2. Каждый чанк шифруется ключом AES-GCM.
  3. Чанки отправляются последовательно в бинарном виде (`ArrayBuffer`).
  4. Получатель собирает чанки во `Blob` и формирует локальную ссылку на скачивание.

🚀 Быстрый запуск и развертывание
Требования
Node.js v18+

Современный браузер с поддержкой Web Crypto API и WebRTC (Chrome, Firefox, Safari, Edge)

1. Клонирование репозитория
Bash
git clone [https://github.com/TheMk-ctrl/nopeer.pw.git](https://github.com/TheMk-ctrl/nopeer.pw.git)
cd nopeer.pw
2. Запуск сигнального сервера
Bash
cd server
npm install
npm start
Сигнальный сервер запустится по адресу http://localhost:9000.

3. Запуск клиента
Для работы Web Crypto API браузер требует безопасный контекст (HTTPS или localhost).

Вы можете запустить клиент с помощью любого статического HTTP-сервера:

Bash
# Из корневой директории
npx serve client
Перейдите в браузере по адресу http://localhost:3000, скопируйте ссылку на комнату и откройте ее во второй вкладке или на другом устройстве для начала зашифрованного P2P-сеанса.
